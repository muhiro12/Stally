# User Experience Principles

## Quiet Accumulation

Stally should feel like a quiet record that becomes useful over time.

The app should not ask the user to enter a lot of detail before value appears.
The first meaningful action is adding an item and marking it once.

## Personal, Not Performative

The product is about the user's own choices.

Language should favor "choose," "reach for," "mark," "history," and
"pattern." Avoid turning the product into a public performance, scorekeeping
system, or generic productivity tracker.

## One Mark Is Enough

Daily capture should stay lightweight.

A single mark communicates that an item was chosen on a day. Do not make the
core workflow depend on counts within the day, ratings, duration, notes,
locations, or other extra measurements unless a later product decision changes
the concept.

## Preserve History

Stally's history is the product's value.

Archive, restore, and edit flows should protect marks rather than casually
discarding them. Destructive actions need clear language.

## Active And Preserved Are Different

Library and Archive express a product distinction:

- Library is for active items.
- Archive is for preserved items that should stay nearby without crowding the
  active list.

Archive is not trash.

## Review Is Gentle Attention

Review should surface items that deserve attention before they drift too far
out of mind.

The experience should avoid guilt. "Needs First Mark," "Dormant," and
"Recovery Candidates" are neutral states that help the user decide a next step.

## Insights Are Pattern Reading

Insights should help the user read the collection as a pattern, not as a
technical analytics dashboard.

Metrics should remain understandable in terms of the collection:

- What was active?
- What stayed quiet?
- Which categories carried marks?
- Where are streaks or gaps?
- Which weekdays or months show rhythm?
- What next move is grounded in the user's own data?

## Safety Belongs Near Risk

Backup Center exists because export, import, replace, and reset actions are
higher-risk than ordinary marking.

Keep risky actions grouped, previewed, and confirmed. Make merge and replace
mean different things in user-facing language.

## Guidance Should Be Contextual And Quiet

Legacy tips introduce main flows only when relevant:

- Begin with one item.
- Open Review when the list starts to drift.
- Handle one lane together.
- Change the reading window first.
- Adjust a past day carefully.
- Export before higher-risk changes.

Guidance should help without becoming a tutorial-heavy experience.

## Context Helps Future Reading

Notes and photos should be optional context that make history easier to read
later. Insights may nudge frequent items without notes, but context should not
be required before the user can begin.

## Empty States Should Be Actionable

Empty states should tell the user what the absence means and offer a clear
next step.

Examples from legacy intent:

- Start with a few pieces you actually reach for.
- Nothing Needs Review.
- No Matching Items.
- No Archived Items.
- No validation issues were found in this backup.

## User Control Over Scope

Insights and review preferences should respect user control.

Preserved controls include:

- Review thresholds for first mark and dormancy.
- Whether completed review sections stay visible.
- Default Insights range.
- Whether archived items are included by default.

The exact settings UI is not preserved, but the product value of adjustable
attention and reading scope is preserved.
